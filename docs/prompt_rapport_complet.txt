SafeContract est une plateforme d'audit de sécurité automatisée pour les smart
contracts Ethereum (Solidity / Vyper), développée dans le cadre d'un projet
d'école d'ingénieurs à l'ESME par trois étudiants :

- Géraud (développement backend + outils d'analyse)

L'objectif est de démocratiser l'audit de sécurité des smart contracts, un
domaine habituellement réservé à des cabinets spécialisés coûteux (30 000 € à
200 000 € par audit). SafeContract offre une analyse automatisée multicouche
combinant outils open-source et intelligence artificielle.

Contexte marché :
- En 2022-2023, plus de 3,8 milliards de dollars ont été volés via des
  vulnérabilités de smart contracts (reentrancy, flash loans, integer overflow).
- Le protocole The DAO (2016) : 60 M$ volés par reentrancy.
- Poly Network (2021) : 600 M$ volés.
- Ronin Bridge (2022) : 625 M$ volés.
- La majorité des projets DeFi ne réalisent pas d'audit avant déploiement.

PIPELINE D'ANALYSE (scan.py) :

Étape 1 — Validation :
- Extension autorisée : .sol ou .vy
- Verrou global asyncio (_scan_lock) : une seule analyse à la fois
- Si verrou occupé → HTTP 409 avec message explicite

Étape 2 — Écriture temporaire :
- tempfile.NamedTemporaryFile() avec suffixe .sol/.vy
- Supprimé à la fin dans un bloc finally

Étape 3 — Exécution parallèle des outils :
```python
with ThreadPoolExecutor(max_workers=5) as executor:
    futures = {executor.submit(_run_tool, name, fn, path): name ...}
```
Outils disponibles selon l'extension et la sélection :
- .sol : mythril, slither, solhint, echidna, foundry
- .vy  : mythril, slither uniquement

Sélection des outils depuis le frontend :
- Le frontend envoie un paramètre `tools` (JSON array) dans le form-data
- Ex: ["slither", "solhint", "ai"] pour analyse statique seule + IA
- None ou absent = tous les outils disponibles

Étape 4 — Agrégation (aggregator.py) :
- normalize_issue() : normalise les champs (title, description, severity,
  line, swcId, tool, confidence) pour chaque issue de chaque outil
- Normalisation de la sévérité : high→critical, informational→low
- _deduplicate() : déduplique par (titre_tronqué[:60], numéro_ligne)
  En cas de doublon, conserve la sévérité la plus haute
- compute_score() : 100 - criticals×30 - mediums×15 - lows×5, min 0

Étape 5 — Analyse GNN (optionnelle) :
Si "ai" dans les outils demandés ET is_available() == True :
- ai_service.analyze_contract(path, issues) est appelé
- Les issues CONFIRMED (outil + GNN d'accord) ont confirmedByGnn=True
- Les issues POTENTIAL (GNN seul, confiance > 80%) sont ajoutées
- Les issues FILTERED (outil signalé, GNN détecte une protection) sont
  notées mais non comptées dans le score
- compute_score_weighted() recalcule le score avec ×1.5 sur les CONFIRMED

Étape 6 — Sauvegarde MongoDB :
```python
doc = {
    "filename": filename,
    "code": code,
    "score": score,
    "issues": issues,        # description + tous les champs GNN
    "summary": summary,
    "tools_used": tools_used,
    "tools_errors": tools_errors,
    "tools_versions": tools_versions,
    "ai_verdict": ai_verdict,
    "raw_tool_results": raw_results,
    "analyzed_at": datetime.utcnow(),
    "status": "completed"
}
```
- Graceful degradation : si MongoDB est indisponible, le scan retourne quand
  même le résultat (erreur loggée, sans id)

OUTILS D'ANALYSE — DÉTAIL :

Mythril (mythril_service.py) :
- Analyse symbolique du bytecode EVM via l'exécutable `myth`
- Utilise un venv dédié (.venv-mythril) pour isoler les dépendances conflictuelles
  (coincurve<21, eth-hash, setuptools==68.2.2 pour pkg_resources sur Python 3.13)
- Commande : myth analyze <file> --execution-timeout 60 --json
- Timeout 180s via signal.alarm
- Détecte : reentrancy, integer overflow, selfdestruct, tx.origin, timestamp, etc.

Slither (slither_service.py) :
- Analyseur statique Python, intégré directement dans le venv principal
- API Python native : Slither(path) plutôt qu'un sous-processus
- Détecteurs : reentrancy, uninitialized-state, arbitrary-send-eth, etc.
- Extrait SWC-ID, numéro de ligne, niveau de confiance

Solhint (solhint_service.py) :
- Linter Solidity (Node.js, installé globalement via npm install -g solhint)
- Appel subprocess : solhint --formatter json <file>
- Catégories : best-practices, security, gas-consumption, style
- Les rules de sécurité sont mappées vers les SWC correspondants

Echidna (echidna_service.py) :
- Fuzzer de smart contracts (Haskell, installé via brew install echidna)
- Génère des séquences de transactions aléatoires
- Les violations sont rapportées en sévérité "medium"
- Timeout 60s, mode coverage

Foundry (foundry_service.py) :
- Framework de test Solidity
- Commandes forge build + forge test
- Extraction des numéros de ligne depuis formattedMessage via regex :
  re.search(r':(\d+):\d+:', fmt_msg)

════════════════════════════════════════════════════════════════════════════════
MODULE GNN — INTELLIGENCE ARTIFICIELLE
════════════════════════════════════════════════════════════════════════════════

C'est la pièce maîtresse différenciante de SafeContract. Le module GNN est
entièrement propriétaire, développé de zéro par l'équipe.

PROBLÈME ADRESSÉ :
Les outils classiques (Mythril, Slither) analysent le code de façon linéaire
ou symbolique mais peinent à comprendre le contexte structurel :
- Mythril génère des faux positifs sur les patterns CEI bien implémentés
- Slither signale des reentrancies protégées par ReentrancyGuard
Le GNN analyse le graphe de flux de contrôle (CFG) pour valider ou invalider
les alertes des outils.

ARCHITECTURE DU MODÈLE (SmartContractGNN) :

```
Entrée : vecteur de 786 dimensions par nœud du CFG
  = 768 dim CodeBERT (représentation sémantique du code)
  + 18 dim features expertes (règles binaires hand-crafted)

Couche 0 : LayerNorm(786)
Couche 1 : GATConv(786 → 128, heads=4, concat=True) → 512 dim
           + LayerNorm(512) + ELU + Dropout(0.2)
Couche 2 : GATConv(512 → 128, heads=1, concat=False) → 128 dim
           + LayerNorm(128)
Skip connection : Linear(786 → 128) [connexion résiduelle]
Fusion : ELU(x_graphe + identite_brute) + Dropout(0.2)
Classifieur : Linear(128 → 2) [vulnérable / sain]
```

Justification des choix architecturaux :
- GATConv (Graph Attention Network) : les têtes d'attention permettent au modèle
  de pondérer l'importance des nœuds voisins (ex. : un nœud require() est plus
  important qu'un nœud de commentaire pour évaluer la sécurité d'un call)
- Connexion résiduelle (skip connection) : évite la disparition du gradient
  dans les réseaux profonds et préserve l'identité locale du nœud
- LayerNorm plutôt que BatchNorm : adapté aux graphes de tailles variables
- ELU plutôt que ReLU : continuité en 0 et gradients négatifs pour mieux
  représenter les absences de patterns

FEATURES EXPERTES (18 dimensions) :
Les 18 dimensions binaires capturent des patterns de sécurité connus :
0  : présence de call.value / send() / transfer()
1  : présence de msg.sender
2  : présence de require / assert / revert
3  : présence de selfdestruct
4  : présence de delegatecall
5-8: (réservé V7)
9  : présence de block.timestamp / now
10 : (réservé)
11 : opérations arithmétiques (+, -, *, /)
12 : boucles (for, while)
13-15 : (réservé)
16 : présence de ecrecover / verify()
17 : présence de nonce / usedSignatures / executed[]

EXTRACTION DU CFG (live_extractor.py) :
- Utilise Slither en API Python pour construire le CFG
- Pour chaque fonction du contrat :
  - Extrait les nœuds du CFG (type + contenu)
  - Extrait les arêtes (flux de contrôle)
- Gestion des imports manquants : génération de stubs Solidity synthétiques
  via inférence de types (adresse, bool, uint256, bytes)
- Format de sortie JSON :
  ```json
  {
    "NomContrat": {
      "graphe_noeuds": [{"type": "...", "contenu": "..."}],
      "aretes": [[0, 1], [1, 2], ...]
    }
  }
  ```

VECTORISATION (live_vectorizer.py) :
- Charge CodeBERT (microsoft/codebert-base, 125M paramètres)
- Pour chaque nœud : tokenisation (max_length=128) + mean pooling → 768 dim
- Concaténation avec le vecteur expert 18 dim → 786 dim
- Sauvegarde en .pt (tenseur PyTorch)

PRÉDICTION ET FILTRE TOPOLOGIQUE (predict.py) :
La logique de prédiction comporte deux couches de filtrage :

Couche 1 — Filtre opérations dangereuses :
Ne traiter que les nœuds contenant une opération à risque :
- .call{, msg.sender.call, .send(, .transfer(
- ecrecover, selfdestruct, delegatecall, tx.origin
- block.timestamp, blockhash, block.difficulty, block.prevrandao
- unchecked, swapExactTokens, latestRoundData

Couche 2 — BFS arrière/avant sur le CFG :
- Reentrancy : BFS arrière → si un ancêtre contient nonReentrant,
  _status != 2, ReentrancyGuard → alerte FILTRÉE
- Signature Replay : BFS → si nonce / usedSignatures / executed[] présent
  → alerte FILTRÉE
- ecrecover : BFS avant → vérification address(0) présente → FILTRÉE

Niveaux de confiance du rapport :
- CONFIRMED  : au moins 1 outil ET GNN concordent → vulnérabilité certaine
- POTENTIAL  : GNN seul avec confiance ≥ 80% → possible vulnérabilité
- FILTERED   : outil a signalé MAIS GNN détecte une protection → faux positif

Vulnérabilités couvertes :
- SWC-107 : Reentrancy
- SWC-115 : Authorization via tx.origin
- SWC-122 : Lack of Signature Verification
- SWC-116 : Timestamp Dependence
- SWC-120 : Weak Randomness
- SWC-101 : Integer Overflow/Underflow
- SWC-113 : Denial of Service
- SWC-106 : Unprotected SELFDESTRUCT
- SWC-112 : Delegatecall to Untrusted Callee
- SWC-104 : Unchecked Return Value
- Unprotected ETH Withdrawal

VERSION DU MODÈLE :
- gnn_smart_contracts_v6_retrain.pth (poids entraînés, figés)
- DIM_TOTALE=786, HEADS=4, SEUIL_DEFAUT=0.60

DÉFIS TECHNIQUES ET SOLUTIONS
════════════════════════════════════════════════════════════════════════════════

1. ISOLATION DE MYTHRIL (Python 3.13 + setuptools)
   Problème : Mythril dépend de pkg_resources qui n'est plus exposé comme
   module top-level dans setuptools ≥ 69 sur Python 3.13.
   Solution : venv dédié .venv-mythril avec setuptools==68.2.2 + coincurve<21.
   Le chemin vers l'exécutable myth est passé via la variable d'environnement
   MYTH_BINARY dans ecosystem.config.cjs.

2. CONFLIT DE DÉPENDANCES PYTHON
   Problème : Mythril, Slither, torch-geometric, weasyprint ont des dépendances
   incompatibles si installées ensemble (eth-hash, coincurve, abi versions).
   Solution : installation en groupes séquentiels, mythril isolé dans .venv-mythril,
   sans requirements.txt global (graph de dépendances trop complexe).

3. PROXY ZUPLO ET RÉPONSES NON-JSON
   Problème : les erreurs FastAPI en debug renvoient du HTML, crashant le
   JSON.parse() côté frontend.
   Solution : lecture systématique en texte (await res.text()) avant tentative
   de JSON.parse(), avec message d'erreur explicite si HTML reçu.

4. SCAN CONCURRENT
   Problème : lancer deux analyses en parallèle sur le même serveur VPS
   (mémoire limitée) provoquait des crashes.
   Solution : asyncio.Lock() global + endpoint GET /scan/status pour que le
   frontend sache si un scan est en cours et affiche un indicateur.

5. SORT DE L'HISTORIQUE PAR DATE
   Problème : tri alphabétique sur les chaînes de dates localisées (ex. "31 mars"
   vs "3 avril") donnait un ordre incorrect.
   Solution : ajout du champ rawDate (ISO 8601) dans la timeline,
   tri via new Date(b.rawDate).getTime() - new Date(a.rawDate).getTime().

6. PAGINATION DE L'HISTORIQUE
   Problème : avec de nombreuses analyses, la liste s'étendait indéfiniment.
   Solution : pagination côté client (HIST_PER_PAGE=10, état histPage),
   reset automatique à la page 1 lors du changement d'onglet.

7. IMPORT MANQUANTS DANS LE CFG (GNN)
   Problème : les contrats importent des interfaces externes (OpenZeppelin, etc.)
   non présentes dans le fichier analysé, bloquant Slither.
   Solution : live_extractor.py génère des stubs Solidity synthétiques par
   inférence de types sur les arguments d'appels.

8. WEASYPRINT SUR LINUX (libpango)
   Problème : WeasyPrint nécessite des bibliothèques système (libpango, harfbuzz).
   Solution : sudo apt install -y libpango-1.0-0 libharfbuzz0b libpangoft2-1.0-0
   dans le script de déploiement.

9. SET<STRING> SPREAD EN TYPESCRIPT
   Problème : [...new Set(array)] cause une erreur TypeScript strict.
   Solution : Array.from(new Set(array))

10. NOMMAGE DES CONTRATS COLLÉS
    Problème : la regex /contract\s+(\w+)/ matchait les commentaires
    (ex. "// Define contract name" → nom = "name")
    Solution : /^\s*contract\s+(\w+)/m (mode multiline, ^ = début de ligne)

SÉCURITÉ DE LA PLATEFORME ELLE-MÊME
════════════════════════════════════════════════════════════════════════════════

Mesures implémentées :
- Validation de l'extension (.sol/.vy uniquement) côté backend
- Écriture dans tempfile (répertoire /tmp, nettoyage garanti en finally)
- Verrou global scan (protection contre les attaques par saturation)
- Pas de persistance du code source en dehors de MongoDB (chiffrement à envisager)
- CORS configuré dans FastAPI (origines autorisées limitées)
- Headers de sécurité via Caddy (HSTS, X-Frame-Options)

Limitations actuelles (à adresser en v2) :
- Authentification frontend hardcodée (admin/admin) → doit être remplacée
  par JWT + OAuth réel (GitHub, Google)
- Pas de rate limiting par utilisateur sur les scans
- Le code source du contrat est stocké en clair en MongoDB
- Pas de sandbox pour l'exécution des outils d'analyse
- Pas de validation antivirale sur les fichiers uploadés

════════════════════════════════════════════════════════════════════════════════
PERFORMANCES ET SCALABILITÉ
════════════════════════════════════════════════════════════════════════════════

Temps d'analyse typiques (VPS 2 vCPU / 4 Go RAM) :
- Slither seul : 3–8 secondes
- Solhint seul : < 1 seconde
- Mythril (60s timeout) : 60–90 secondes
- Echidna (60s timeout) : 60–70 secondes
- Foundry : 5–15 secondes
- GNN : 20–45 secondes (dépend de la taille du contrat + CodeBERT)
- Pipeline complet : 90–180 secondes

Optimisations en place :
- Exécution parallèle des outils (ThreadPoolExecutor, max_workers=5)
- Précalcul LLM unique (_precalculer_llm) pour éviter les doubles appels
- Cache des résultats GNN par session (pas de ré-analyse à chaque requête PDF)
- Index MongoDB sur analyzed_at (tri historique)
- Streaming du scan status via polling (GET /scan/status) côté frontend

Limites de scalabilité :
- Un seul scan à la fois (verrou global)
- GNN charge CodeBERT en mémoire pour chaque analyse (~500 Mo VRAM/RAM)
- Mythril est CPU-intensif (analyse symbolique complète du bytecode)
- Pour passer à l'échelle : worker queue (Celery + Redis), instances multiples
